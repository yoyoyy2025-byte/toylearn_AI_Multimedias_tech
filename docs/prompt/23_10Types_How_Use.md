## 프롬프트 유형과 선택 이유

모호함 제거 및 구체화: '깔끔한 대시보드', '칼로리 계산' 같은 추상적인 아이디어를 [출력값] 템플릿의 'F-03: 리포트 출력', 'P-04: 대시보드' 등 구체적인 기능(F)과 페이지(P) 단위로 강제 변환시킵니다. 이는 개발 리더의 "구조를 잡을 수 없다"는 피드백을 정면으로 해결하는 방식입니다.

명확한 산출물 정의: AI가 창의적으로 보고서 형식을 고민하게 두는 대신, PM님이 원하는 정확한 목차와 구조([출력값])를 제시했습니다. 이는 개발팀과 디자인팀이 실제 업무에서 사용하는 '요구사항 정의서(IRD)' 형식과 일치하므로, AI가 생성한 결과물을 거의 그대로 실무에 사용할 수 있게 만듭니다.

트렌드 반영 및 역할 부여: '최신 트렌드를 선도하는 주니어 PM'이라는 페르소나는, AI가 단순히 기능만 나열하는 것이 아니라 **최신 헬스케어 트렌드(예: AI 기반 식단 분석, 게이미피케이션, 하이퍼-퍼스널라이제이션)**를 고려하여 빈칸을 채우도록 유도합니다. [출력값] 템플릿의 예시(AI가 칼로리와 영양소를 분석)에서 이미 이 의도가 잘 드러나 있습니다.

## 작성 위해 추가 필요 요건


1. [핵심] 타깃 페르소나 (Target Persona): '개인 맞춤형'의 '개인'은 누구인가요?

A안: "체중 감량과 바디 프로필이 목적인 2030 직장인" (이 경우, 칼로리와 탄/단/지 매크로 밸런스가 가장 중요)

B안: "건강 관리가 필요한 4050 중장년층" (이 경우, 칼로리보다는 콜레스테롤, 나트륨, 혈당 관리 등 영양소의 질이 중요)

필요 이유: 이 결정에 따라 F-03(리포트)와 P-04(대시보드)에 강조되어야 할 핵심 지표가 완전히 달라집니다.

2. [핵심] 식단 입력 방식 (Meal Logging Method): 사용자가 '어떻게' 입력하나요?

A안 (Basic MVP): 사용자가 '검색' 또는 '수동'으로 입력 (개발이 가장 빠름)

B안 (Trend MVP): AI '사진' 인식 (+ 수동 수정) (트렌디하지만 AI 모델 연동 필요)

C안 (Utility MVP): '바코드' 스캔 (+ 수동 수정) (편의성은 높으나 DB 의존도 높음)

필요 이유: B안이나 C안을 택할 경우, [내용]에 없던 'AI 분석' 또는 '바코드 스캐너' 기능이 명세서에 추가되어야 하며, 백엔드와 프론트엔드의 작업 범위가 완전히 달라집니다.

3. '최소 기능(MVP)'의 범위 (Scope of 'Minimum'): 무엇을 과감히 '제외'할 것인가요?

운동 기록: 식단과 긴밀하지만, MVP에서는 제외하고 '목표 칼로리'만 사용자가 직접 설정하게 할까요?

커뮤니티/소셜 기능: MVP 범위에서 100% 제외하는 것이 맞을까요? (아마도 맞습니다)

필요 이유: '최소' 기능에 집중해야 한다는 점을 명확히 하여, 개발팀이 핵심(식단 기록/리포트)에만 집중하도록 가이드해야 합니다.

4. 핵심 데이터 소스 (Data Source): 식품 영양 정보는 어디서 가져오나요?

A안: 공공 데이터 API 활용 (예: 식품의약품안전처 DB) (무료, 데이터 품질이 낮거나 업데이트가 느릴 수 있음)

B안: 유료 솔루션 API 연동 (예: FatSecret) (비용 발생, 고품질의 글로벌 데이터)

C안: 자체 DB 구축 (운영 리소스 필요, MVP에서는 비추천)

필요 이유: 이는 NF-01(성능)과 F-02(식단 입력)의 구현 방식, 그리고 프로젝트 예산에 직접적인 영향을 줍니다.

## 프롬프트 실행 결과

안녕하세요! 헬스케어 주니어 PM입니다.
